# Crate vs BastilleBSD — Детальне порівняння

## Загальний огляд

| | **Crate** | **BastilleBSD** |
|---|---|---|
| **Призначення** | Контейнеризатор FreeBSD — пакує застосунки та сервіси в ізольовані самодостатні «крейти» | Система автоматизації розгортання та керування контейнеризованими застосунками на FreeBSD |
| **Мова** | C++17 (~3 066 рядків) | Shell-скрипти (POSIX sh) |
| **Ліцензія** | BSD 3-Clause | BSD |
| **Статус** | Alpha (з червня 2019) | Стабільний (v1.3.x, активна розробка з 2018) |
| **Філософія** | Мінімалістичні одноразові контейнери для застосунків; акцент на десктопні застосунки та GUI | Повнофункціональний менеджер jail-ів для серверів та інфраструктури; DevOps-орієнтований |
| **Залежності** | yaml-cpp, libjail, librang | Лише базова система FreeBSD (sh, jail, zfs, pkg) |

---

## Архітектура та модель роботи

### Crate — «Пакування та запуск»
```
spec.yml → [crate create] → myapp.crate (XZ-архів)
                                  ↓
                            [crate run]
                                  ↓
                    Тимчасовий jail + мережа + монтування
                                  ↓
                          Виконання застосунку
                                  ↓
                    Повне очищення (jail знищується)
```
- **Ефемерна модель**: jail створюється під час запуску та знищується після завершення
- Дві команди: `create` та `run`
- Агресивна оптимізація: аналіз ELF-залежностей через ldd, видалення непотрібних файлів
- Контейнер — самодостатній XZ-архів (`.crate`)

### BastilleBSD — «Створення та керування»
```
bastille bootstrap 14.2-RELEASE → базовий реліз
bastille create myjail 14.2-RELEASE 10.0.0.1 → persistent jail
bastille start/stop/restart myjail → керування життєвим циклом
bastille template myjail user/template → автоматизація налаштування
```
- **Персистентна модель**: jail-и існують довго, керуються як сервіси
- 35+ підкоманд для повного керування життєвим циклом
- Jail-и зберігаються на диску (ZFS або UFS)
- Підтримка тонких (thin) та товстих (thick) jail-ів

## CLI-команди

### Crate (2 команди)
| Команда | Опис |
|---|---|
| `crate create -s spec.yml -o app.crate` | Створити контейнер зі специфікації |
| `crate run -f app.crate [-- args]` | Запустити контейнер |
| `crate spec.yml` | Скорочення для create |
| `crate app.crate` | Скорочення для run |

## Типи jail-ів

| Тип | **Crate** | **BastilleBSD** |
|---|---|---|
| Тонкі (thin/shared base) | Ні — кожен .crate повністю самодостатній | **Так** (за замовчуванням) — спільна базова система через nullfs |
| Товсті (thick/independent) | Фактично так — кожен .crate містить повну копію | **Так** (прапорець `-T`) |
| Клони (clone) | Ні | **Так** (прапорець `-C`, ZFS clone) |
| Порожні (empty) | Ні | **Так** (прапорець `-E`) — для власних збірок |
| Linux jail-и | Ні | **Так** (`-L`, експериментально, через debootstrap) |

## Мережа

| Можливість | **Crate** | **BastilleBSD** |
|---|---|---|
| **VNET** | Так (epair) | Так (3 режими: -V, -B, -P) |
| **Фізичний інтерфейс** | Ні | Так (`-V` — прив'язка до фізичного інтерфейсу) |
| **Bridge** | Ні | Так (`-B` — підключення до наявного мосту) |
| **Passthrough** | Ні | Так (`-P` — прокидання інтерфейсу в jail) |
| **NAT** | Так (ipfw NAT, автоматичні правила) | Через pf/ipfw (ручне налаштування або rdr) |
| **Перенаправлення портів** | Так (inbound-tcp/udp у YAML) | Так (`bastille rdr`) |
| **IP-адресація** | Авто (10.0.0.0/8 на основі ID контейнера) | Ручна (вказується при створенні) |
| **DNS** | Необов'язкова пересилка | Налаштування через прапорець `-n` |
| **Контроль вихідного трафіку** | Так (wan/lan/host/dns гранулярність) | Через правила файрволу (pf/ipfw) |
| **VLAN** | Ні | Так (прапорець `-v`) |
| **Static MAC** | Ні | Так (прапорець `-M`) |
| **IPv6** | Ні | Так (покращено у v1.0+) |
| **Динамічні epair** | Ні (статичні) | Так (з v1.0, обидві форми VNET одночасно) |

**Висновок**: Crate надає зручне автоматичне керування мережею з гранулярним контролем вихідного трафіку. BastilleBSD пропонує значно більше мережевих режимів та конфігурацій корпоративного рівня.

| Можливість | **Crate** | **BastilleBSD** |
|---|---|---|

| **ZFS snapshots** | Ні | **Так** |
| **ZFS clone** | Ні | **Так** (для тонких jail-ів) |
| **ZFS send/recv** | Ні | **Так** (використовується в migrate) |
| **UFS** | Так (за замовчуванням) | **Так** |

## Шаблонізація та автоматизація

### Crate — YAML-специфікація
```yaml
pkg:
    install: [firefox, git]
run:
    command: /usr/local/bin/firefox
options: [net, x11, gl, video]
dirs:
    share:
        - [/var/db/myapp, $HOME/myapp/db]
scripts:
    run:begin: ["echo 'Starting...'"]
    run:after-execute: ["cleanup.sh"]
```
- Єдиний YAML-файл описує все
- 10+ хуків життєвого циклу (run:begin, run:after-create-jail, run:before-start-services тощо)
- Підстановка змінних ($HOME, $USER, ${VAR})

### BastilleBSD — Bastillefile
```
# Bastillefile
CMD echo "Hello"
PKG nginx
SYSRC nginx_enable=YES
SERVICE nginx start
TEMPLATE /usr/local/etc/nginx/nginx.conf
OVERLAY /usr/local/www
```
- Docker-подібний синтаксис
- Команди: CMD, PKG, SYSRC, SERVICE, TEMPLATE, OVERLAY, CP, RDR, MOUNT, CONFIG та ін.
- Шаблони зберігаються у Git-репозиторіях
- Застосовуються до вже створених jail-ів
- Консолідований репозиторій шаблонів на GitLab

**Висновок**: Обидва підходи ефективні, але відрізняються за філософією. Crate описує «що має бути всередині контейнера», BastilleBSD описує «як налаштувати наявний jail».

## Файрвол

| Можливість | **Crate** | **BastilleBSD** |
|---|---|---|
| **Файрвол** | ipfw (автоматичні правила NAT) | **pf** (нативна інтеграція) |

**Висновок**: Crate використовує ipfw з автоматичним керуванням. BastilleBSD орієнтований на pf з повним контролем якорів та таблиць.

## Bastillefile — повний довідник хуків

| Хук | Синтаксис | Призначення |
|------|-----------|-------------|
| `ARG` | `ARG_NAME=default` | Визначення змінної зі значенням за замовчуванням |
| `ARG+` | `ARG_NAME+=value` | Обов'язковий аргумент |
| `CMD` | `/bin/sh command` | Виконання команд у jail |
| `CONFIG` | `set property value` | Встановлення властивостей jail.conf |
| `CP` | `usr etc` | Копіювання overlay-каталогів |
| `INCLUDE` | `user/template` або URL | Включення іншого шаблону |
| `LIMITS` | `resource value` | Обмеження ресурсів (rctl) |
| `LINE_IN_FILE` | `word /path/file` | Умовне додавання рядків до файлу |
| `MOUNT` | `hostpath jailpath nullfs rw 0 0` | Монтування томів |
| `PKG` | `pkg1 pkg2` | Встановлення пакетів |
| `RDR` | `tcp 80 8080` | Перенаправлення портів |
| `RENDER` | `/path/to/file` | Підстановка змінних `${ARG}` |
| `RESTART` | (без аргументів або сервіс) | Перезапуск jail/сервісу |
| `SERVICE` | `servicename start` | Керування сервісами |
| `SYSRC` | `key=value` | Встановлення значень rc.conf |
| `TAGS` | `tag1 tag2` | Призначення міток |

Вбудовані змінні: `${JAIL_NAME}`, `${JAIL_IP}`, `${JAIL_IP6}`

## Керування релізами

| Можливість | **Crate** | **BastilleBSD** |
|---|---|---|
| **Завантаження бази** | FTP (base.txz) | `bootstrap` (freebsd-update або pkgbase) |
| **Оновлення патчів** | Перезбирання .crate | `bastille update` (freebsd-update) |
| **Оновлення релізу** | Перезбирання .crate | `bastille upgrade` (thin та thick) |
| **Оновлення /etc** | Перезбирання .crate | `bastille etcupdate` |
| **Верифікація** | Ні | `bastille verify` |
| **pkgbase (FreeBSD 15+)** | Ні | **Так** (`bootstrap --pkgbase`) |

## Коли що використовувати?

### Crate краще підходить для:
- Запуску **десктопних GUI-застосунків** в ізольованому середовищі (Firefox, Chromium, Kodi)
- **Одноразових** ізольованих середовищ виконання
- Мінімізації розміру контейнера (оптимізація ELF-залежностей)
- Швидкого прототипування ізольованих застосунків
- Сценаріїв, що потребують X11/OpenGL/відео

### BastilleBSD краще підходить для:
- **Серверної інфраструктури** та DevOps
- Керування **численними** довготривалими jail-ами
- Середовищ із **ZFS** (знімки, клони, міграція)
- **Міграції** jail-ів між серверами (включно з live)
- Автоматизації через **шаблони** (CI/CD)
- Гранулярного контролю **ресурсів** (CPU, RAM, I/O)
- Сумісності з іншими менеджерами jail-ів (iocage, ezjail імпорт)
- Роботи з **Linux jail-ами**

## Що Crate міг би запозичити у BastilleBSD

1. **Інтеграція із ZFS** — знімки, клони, швидке створення jail-ів
2. **Персистентний режим** — можливість збереження jail-у між запусками
3. **Перелік/моніторинг** — команда для перегляду запущених контейнерів
4. **Обмеження ресурсів** — rctl/cpuset для обмеження CPU/RAM
5. **Клонування** — створення копій наявних .crate
6. **Тонкі jail-и** — економія місця через спільну базову систему
7. **IPv6** — підтримка IPv6 у мережевому стеку
8. **Bridge/Passthrough** — додаткові мережеві режими
9. **Команда list** — перегляд доступних .crate файлів та запущених контейнерів
10. **Live-міграція** — перенесення працюючого контейнера на інший хост
