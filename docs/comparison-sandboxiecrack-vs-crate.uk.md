# SandboxieCrack vs Crate: Порівняння архітектур

## Огляд

| | SandboxieCrack (Windows) | Crate (FreeBSD) |
|---|---|---|
| **Платформа** | Windows (ядро NT) | FreeBSD |
| **Мова** | C (драйвер/dll), C++ (сервіси/UI) | C++17 |
| **Зрілість** | Production, 20+ років розробки | Alpha |
| **Ліцензія** | GPL-3.0 | GPL-3.0 |
| **Upstream** | sandboxie-plus/Sandboxie | yurivict/crate |
| **Архітектура** | 4 рівні: драйвер ядра → DLL-хуки → сервіси → GUI | Один setuid-бінарник, що керує FreeBSD jail |

## Архітектурні моделі

### SandboxieCrack — Ізоляція на основі перехоплення

SandboxieCrack використовує багаторівневий підхід перехоплення у Windows:

1. **Драйвер ядра** (`core/drv/`) — Перехоплює 50+ системних викликів NT через SSDT-хуки та minifilter-callback'и. Реалізує трансляцію шляхів файлів (`file.c`, `file_xlat.c`), ізоляцію реєстру через hive-файли (`key.c`, `key_flt.c`), маніпуляцію токенами процесів через `Token_Filter`/`Token_Restrict`/`Token_ReplacePrimary` (`token.c`), фільтрацію IPC з динамічним відстеженням портів через структури `IPC_DYNAMIC_PORT` (`ipc.c`, `ipc_port.c`) та мережеву фільтрацію на базі WFP з підрівнем для кожної пісочниці та відстеженням потоків (`wfp.c`). Драйвер зберігає стан кожної пісочниці через структуру `BOX` (`box.h`), яка визначає сім шляхів просторів імен: файловий (`\??\C:\Sandbox\%SID%\BoxName`), raw file (варіант з reparse-point), реєстр (`HKCU\Sandbox\BoxName`), IPC (`\Sandbox\%SID%\Session_%SESSION%\BoxName`), pipe (шлях IPC з підкресленнями замість зворотних слешів), spooler directory (ізоляція черги друку) та temp-шляхи (`system_temp_path`, `user_temp_path` для ізоляції тимчасових файлів кожної пісочниці).

2. **Користувацька DLL** (`core/dll/`) — Впроваджується в кожен процес пісочниці через низькорівневий хук `LdrInitializeThunk`. Найбільші компоненти: `file.c` (258КБ — COW-операції з файлами з відстеженням дерева видалень), `key.c` (144КБ — перехоплення реєстру зі злиттям значень через `key_merge.c`), `com.c` (101КБ — перехоплення активації COM/DCOM з проксі-обгортками `COM_IUNKNOWN`), `proc.c` (100КБ — перехоплення `CreateProcessW`/`ShellExecuteEx` для перенаправлення створення процесів), `gui.c` (87КБ — ізоляція вікон/робочих столів/буфера обміну), `net.c` (72КБ — перехоплення мережевих API сокетів), `dns_filter.c` (25КБ — фільтрація DNS-розв'язання з кешуванням IP-записів). Управління хуками через структури `MODULE_HOOK` у `dllhook.c` з detour-трамплінами.

3. **Рівень сервісів** (`core/svc/`) — Привілейований Windows-сервіс зі спеціалізованими серверами: `ProcessServer` (створення процесів, управління DACL токенів, запити RunSandboxed), `MountManager` (1455 рядків — повне управління віртуальними дисками: `MountImDisk` створює шифровані образи томів з AES-шифром та захистом паролем або RAM-диски через `GetRamDisk`; `UnmountImDisk` виконує демонтаж з очищенням; форматування нових томів у NTFS; призначення літер дисків; управління NTFS-junction через `CreateJunction`/`RemoveJunction`; блокування кореня пісочниці через `AcquireBoxRoot`/`ReleaseBoxRoot`; зберігання паролів у пам'яті для запобігання потрапляння ключів на диск), `GuiServer` (проксі вікон з архітектурою slave-процесів для міжпісочнічної GUI-медіації), та `NamedPipeServer` (проксі іменованих каналів). Комунікація через заголовки wire-протоколу (`*Wire.h`).

4. **GUI** (`SandboxiePlus/`) — Qt-інтерфейс управління з конфігурацією пісочниць, моніторингом та управлінням знімками.

### Crate — Ізоляція на рівні ОС через jail

Crate повністю делегує ізоляцію ядру FreeBSD:

1. **Один бінарник** — Setuid C++ програма, що керує життєвим циклом jail: створює образи контейнерів з базової FreeBSD + пакети, запускає їх як jail з VNET-мережею.

2. **Jail-ізоляція** — Використовує `jail_setv()` для створення jail з ізоляцією процесів, файлової системи та мережі. Не потребує перехоплення API — ядро забезпечує межі.

3. **VNET-мережа** — Створює віртуальні інтерфейси `epair`, призначає IP-адреси (схема 10.x.y.z) та налаштовує правила `ipfw` NAT для вихідного з'єднання та опціонального перенаправлення вхідних портів.

4. **Файлова система** — Використовує `nullfs` bind-mount'и для спільних директорій/файлів та `devfs` для доступу до пристроїв усередині jail. Образи контейнерів — стиснуті tar-архіви (xz).

## Порівняння функцій

| Аспект | SandboxieCrack (Windows) | Crate (FreeBSD) |
|--------|--------------------------|------------------|
| **Базова ізоляція** | Драйвер ядра (`drv/`) + DLL-хуки (`dll/`) — перехоплення syscall | FreeBSD jail через `jail_setv()` — ізоляція на рівні ядра |
| **Віртуалізація файлів** | Трансляція шляхів + copy-on-write (`file.c`, `file_xlat.c`) з семантикою TRUE_PATH/COPY_PATH | Окрема rootfs (tar-архів) + `nullfs` bind-mount для спільного доступу |
| **Ізоляція реєстру/конфігурації** | Перенаправлення hive реєстру (`key.c`, `key_flt.c`) з фільтрацією ключів; злиття реєстру через `key_merge.c` реалізує COPY_PATH-семантику для значень реєстру | Н/З — у FreeBSD немає реєстру; per-jail `/etc` достатньо |
| **Мережева ізоляція** | WFP попроцесна фільтрація (`wfp.c`) + DNS-перехоплення (`dns_filter.c`) | VNET + віртуальні інтерфейси `epair` + `ipfw` NAT (`net.cpp`, `run.cpp`) |
| **Контроль процесів** | Маніпуляція токенами + зниження привілеїв (`token.c`, `process.c`), рівні цілісності | Ізоляція процесів на рівні jail; `jexec` для виконання всередині jail |
| **Ізоляція IPC** | Фільтрація іменованих каналів/ALPC (`ipc.c`, `ipc_port.c`), блокування динамічних портів | Ізоляція простору імен IPC jail (`allow.sysvipc`) |
| **Ізоляція GUI** | Фільтрація класів вікон, перехоплення буфера обміну (`gui.c`, `guihook.c`) | Спільний X11-сокет через `nullfs` mount (без ізоляції) |
| **Перехоплення API** | DLL-ін'єкція + IAT/inline хуки (`dllhook.c`, `hook_tramp.c`) | Не потрібно — jail-модель не вимагає хуків на рівні API |
| **Ізоляція криптографічного API** | Перехоплення DPAPI (`crypt.c`) — перехоплює `CryptProtectData`/`CryptUnprotectData` для ізоляції сховища облікових даних | Не потрібно — per-jail домашня директорія ізолює файли облікових даних |
| **Шифровані пісочниці** | Шифровані томи ImBox через `MountManager.cpp` — AES-шифр, підтримує як image-файли (`type=img`), так і RAM-диски (`type=ram`), форматування NTFS, призначення літер дисків | Не реалізовано |
| **Знімки** | Файлові знімки з шифруванням імен (`file_snapshots.c`) | Не реалізовано |
| **Ліміти ресурсів** | Ліміт кількості процесів на пісочницю | Не реалізовано (доступний фреймворк RCTL у FreeBSD) |
| **Рівень сервісів** | C++ сервіси з wire-протоколом (`svc/`) — брокеринг процесів, COM-проксі, GUI-сервер | Відсутній — один бінарник обробляє все |
| **Шаблони/Пресети** | Типи пісочниць: Standard, Enhanced Privacy, Data Protection, Application Compartment | Відсутні — ручний YAML для кожного контейнера |
| **Конфігурація** | INI (`Sandboxie.ini`) з GUI-редактором та валідацією | YAML (`+CRATE.SPEC`) з програмним парсингом |
| **Управління пакетами** | Н/З (запускає програми хоста) | Інтеграція з FreeBSD `pkg` — встановлює пакети в образ контейнера |
| **COM/D-Bus сервіси** | Перехоплення COM/DCOM (`com.c`, 101КБ) — перехоплення `CoCreateInstance`, перенаправлення DCOM, проксі `COM_IUNKNOWN` | Не реалізовано (можливий D-Bus per-jail) |
| **Ізоляція буфера обміну** | Перехоплення буфера обміну через GUI-хуки + міжпісочнічна медіація `GuiServer` | Не ізольовано — спільний через X11-сокет |
| **Контроль створення процесів** | Хуки `CreateProcessW`/`ShellExecuteEx` (`proc.c`, 100КБ) — перенаправлення дочірніх процесів у пісочницю | Нативно через jail — `jexec` для входу, межа jail забезпечена ядром |
| **Перехоплення syscall** | SSDT-хуки 50+ syscall NT на рівні ядра | Не потрібно — FreeBSD jail забезпечують ізоляцію на рівні ядра без перехоплення syscall |
| **Ізоляція SCM/сервісів** | Перехоплення Service Control Manager (`scm.c`, `scm_*.c`, `serviceserver.cpp`) — програми у пісочниці перелічують/запускають сервіси в контексті пісочниці; доступ до реальних сервісів хоста заблоковано | Per-jail `rc.d` сервіси; ізоляція сервісів jail є нативною |
| **Проксі іменованих каналів/сокетів** | Проксі-сервер іменованих каналів (`namedpipeserver.cpp`, 853 рядки) + перехоплення pipe (`file_pipe.c`) — проксіює pipe через межу пісочниці | Unix domain сокети; `nullfs`-mount окремих сокетів для контрольованого спільного доступу |
| **Ізоляція терміналу/консолі** | Перехоплення API терміналу (`terminal.c`) + привілейований сервер терміналу (`terminalserver.cpp`) — ізолює вікна консолі для кожної пісочниці | Per-jail виділення PTY через `devfs`-правила для `/dev/pts` |
| **Проксі мережевих API** | Проксіювання API мережевої інформації (`netapiserver.cpp`, `iphlpserver.cpp`) — контролює перелічення мережевих ресурсів та запити IP helper | VNET забезпечує нативну per-jail ізоляцію мережевої інформації |
| **Захищене сховище** | Проксі-сервер PStore (`pstoreserver.cpp`) — ізолює збережені облікові дані та секрети для кожної пісочниці | Per-jail домашня директорія; без спільного сховища облікових даних |
| **Портативність контейнерів** | Пісочниці прив'язані до інсталяції хоста | Стиснуті tar-архіви (`.crate` файли) — розповсюджувані образи |

## Аналіз прогалин: Чого не вистачає Crate

Функції, наявні в SandboxieCrack, але відсутні в Crate:

1. **Шифровані контейнери** — SandboxieCrack надає шифровані томи ImBox з AES; Crate не має підтримки шифрування.

2. **Управління знімками** — SandboxieCrack підтримує файлові знімки з ланцюжками батько-нащадок; Crate не має механізму знімків.

3. **Ліміти ресурсів контейнера** — SandboxieCrack обмежує кількість процесів на пісочницю; Crate не забезпечує обмеження ресурсів (хоча фреймворк RCTL FreeBSD міг би це забезпечити).

4. **DNS-фільтрація** — SandboxieCrack перехоплює DNS-розв'язання для кожної пісочниці з підтримкою шаблонів доменів; Crate лише налаштовує базову NAT-мережу.

5. **Шаблони конфігурації** — SandboxieCrack пропонує попередньо визначені профілі безпеки (Privacy, Data Protection тощо); Crate вимагає ручного написання YAML.

6. **GUI-управління** — SandboxieCrack має повноцінний Qt GUI для управління пісочницями; Crate працює лише через CLI.

7. **Валідація конфігурації** — SandboxieCrack валідує налаштування з контекстними підказками; Crate має лише базову валідацію специфікації.

8. **Copy-on-write файлова система** — SandboxieCrack прозоро перенаправляє записи файлів у копії пісочниці; Crate використовує повні копії rootfs без COW.

9. **Ізоляція GUI/робочого стола** — SandboxieCrack фільтрує віконні операції, ізолює робочі столи та посередничає в міжпісочнічному GUI-доступі через GuiServer; Crate надає спільний X11-сокет хоста без будь-якої ізоляції.

10. **Ізоляція буфера обміну** — SandboxieCrack забезпечує буфери обміну для кожної пісочниці з налаштовуваними політиками міжпісочнічної вставки; Crate не має ізоляції буфера обміну.

11. **Ізоляція COM/D-Bus сервісів** — SandboxieCrack перехоплює активацію COM та проксіює DCOM-об'єкти; Crate не має ізоляції D-Bus або сервісної шини.

12. **Ізоляція SCM/менеджера сервісів** — SandboxieCrack перехоплює Windows Service Control Manager, щоб програми у пісочниці могли перелічувати та керувати сервісами в контексті пісочниці; Crate покладається на нативну ізоляцію jail-сервісів.

13. **Ізоляція терміналу/консолі** — SandboxieCrack ізолює вікна консолі та термінальні сеанси для кожної пісочниці з виділеним сервером терміналу; Crate має базовий per-jail PTY, але без явного управління сеансами терміналу.

14. **Проксі іменованих каналів / Unix-сокетів** — SandboxieCrack має виділений проксі-сервер іменованих каналів (853 рядки) для контрольованої міжпісочнічної комунікації через pipe; Crate не має еквівалентного механізму проксі Unix domain сокетів.

## Сильні сторони кожного підходу

### Сильні сторони SandboxieCrack

- **Прозорість для програм** — Програми у пісочниці працюють без змін; перехоплення API робить пісочницю невидимою для програм.
- **Тонкий контроль** — Політики для кожного процесу, файлу, ключа реєстру. Можна дозволити/заборонити окремі виклики API.
- **Без дублювання rootfs** — Спільна ОС хоста; копіюються лише змінені файли (copy-on-write). Легкий за дисковим простором.
- **Зрілий набір функцій** — 20+ років розробки. Шифрування, знімки, ліміти ресурсів, DNS-фільтрація, COM-проксі.
- **Глибина інтеграції** — Глибока інтеграція з Windows: контекстні меню оболонки, опція "Запустити у пісочниці", автозапуск програм.

### Сильні сторони Crate

- **Ізоляція на рівні ядра** — Jail — це примітив ядра. Немає хуків у просторі користувача для обходу — сильніша межа безпеки.
- **Повна ізоляція мережевого стека** — VNET надає кожному контейнеру власний мережевий стек, таблицю маршрутизації та файрвол. SandboxieCrack лише фільтрує на рівні WFP.
- **Портативність** — Образи контейнерів (`.crate` файли) — самодостатні tar-архіви. Легко розповсюджувати та відтворювати.
- **Простота** — Один бінарник, ~2000 рядків C++. Легко аудитувати та аналізувати властивості безпеки.
- **Нативне управління пакетами** — Використовує екосистему FreeBSD `pkg` для побудови образів контейнерів з конкретним програмним забезпеченням.
- **Хуки скриптів** — YAML-специфікація підтримує shell-скрипти у 10 точках життєвого циклу (до/після створення jail, налаштування користувачів, запуску сервісів тощо).
- **Не потребує модуля ядра** — Використовує лише стандартні засоби FreeBSD jail/VNET/ipfw. Без кастомного коду ядра.
- **Екосистема, готова до ZFS** — Підтримка ZFS у FreeBSD дозволяє потенційно реалізувати COW, знімки та шифрування без змін у програмах.
