# SandboxieCrack проти Crate: Порівняння архітектур

## Огляд

| | SandboxieCrack (Windows) | Crate (FreeBSD) |
|---|---|---|
| **Платформа** | Windows (ядро NT) | FreeBSD |
| **Мова** | C (драйвер/DLL), C++ (сервіси/інтерфейс) | C++17 |
| **Зрілість** | Продакшн, 20+ років розробки | Альфа |
| **Ліцензія** | GPL-3.0 | GPL-3.0 |
| **Репозиторій** | sandboxie-plus/Sandboxie | yurivict/crate |
| **Архітектура** | 4 рівні: драйвер ядра → DLL-перехоплення → сервіси → GUI | Один setuid-бінарник, що керує jail-ами FreeBSD |

## Архітектурні моделі

### SandboxieCrack — ізоляція на основі перехоплення

SandboxieCrack використовує багаторівневий підхід перехоплення у Windows:

1. **Драйвер ядра** (`core/drv/`) — Перехоплює 50+ системних викликів NT через SSDT-перехоплення та зворотні виклики мініфільтрів. Реалізує трансляцію файлових шляхів (`file.c`, `file_xlat.c`), ізоляцію гілок реєстру (`key.c`, `key_flt.c`), маніпуляцію токенами процесів через `Token_Filter`/`Token_Restrict`/`Token_ReplacePrimary` (`token.c`), фільтрацію IPC з динамічним відстеженням портів через структури `IPC_DYNAMIC_PORT` (`ipc.c`, `ipc_port.c`) та мережеву фільтрацію на основі WFP з окремим підрівнем для кожної пісочниці та відстеженням потоків (`wfp.c`). Драйвер зберігає стан кожної пісочниці через структуру `BOX` (`box.h`), яка визначає сім шляхів просторів імен: файловий (`\??\C:\Sandbox\%SID%\BoxName`), сирий файловий (варіант з reparse-point), реєстр (`HKCU\Sandbox\BoxName`), IPC (`\Sandbox\%SID%\Session_%SESSION%\BoxName`), канал (IPC-шлях зі зворотними скісними рисками як підкресленнями), каталог спулера (ізоляція черги друку) та тимчасові шляхи (`system_temp_path`, `user_temp_path` для ізоляції тимчасових файлів кожної пісочниці).

2. **DLL користувацького режиму** (`core/dll/`) — Впроваджується в кожен ізольований процес через низькорівневе перехоплення `LdrInitializeThunk`. Найбільші компоненти: `file.c` (258 КБ — файлові операції copy-on-write з відстеженням дерева видалень), `key.c` (144 КБ — перехоплення реєстру зі злиттям значень через `key_merge.c`), `com.c` (101 КБ — перехоплення активації COM/DCOM з проксі-обгортками `COM_IUNKNOWN`), `proc.c` (100 КБ — перехоплення `CreateProcessW`/`ShellExecuteEx` для перенаправлення створення процесів), `gui.c` (87 КБ — ізоляція вікон/робочих столів/буфера обміну), `net.c` (72 КБ — перехоплення сокетів та мережевого API) та `dns_filter.c` (25 КБ — фільтрація DNS-розв'язання з кешуванням IP-записів). Управління перехопленнями через структури `MODULE_HOOK` у `dllhook.c` з трамплінами в стилі detour.

3. **Рівень сервісів** (`core/svc/`) — Привілейований Windows-сервіс зі спеціалізованими серверами: `ProcessServer` (створення процесів, управління DACL токенів, запити RunSandboxed), `MountManager` (1455 рядків — повне управління віртуальними дисками: `MountImDisk` створює зашифровані томи на основі образів з AES-шифруванням та захистом паролем або тимчасові диски в оперативній пам'яті через `GetRamDisk`; `UnmountImDisk` відповідає за демонтаж з очищенням; форматування нових томів у NTFS; призначення та управління літерами дисків; управління NTFS-з'єднаннями через `CreateJunction`/`RemoveJunction`; блокування кореня пісочниці через `AcquireBoxRoot`/`ReleaseBoxRoot`; зберігання паролів у пам'яті для запобігання витоку ключів на диск), `GuiServer` (проксі вікон з архітектурою підлеглих процесів для посередництва GUI між пісочницями) та `NamedPipeServer` (проксіювання каналів). Комунікація через заголовки дротового протоколу (`*Wire.h`).

4. **GUI** (`SandboxiePlus/`) — Інтерфейс управління на базі Qt з налаштуванням пісочниць, моніторингом та управлінням знімками.

### Crate — ізоляція на рівні ОС через jail

Crate повністю делегує ізоляцію ядру FreeBSD:

1. **Один бінарник** — Setuid-програма на C++, яка керує життєвим циклом jail: створює образи контейнерів з базової системи FreeBSD + пакетів, запускає їх як jail з мережевою підсистемою VNET.

2. **Ізоляція через jail** — Використовує `jail_setv()` для створення jail-ів з ізоляцією процесів, файлової системи та мережевого простору імен. Перехоплення API не потрібне — ядро само забезпечує межі ізоляції.

3. **Мережева підсистема VNET** — Створює віртуальні інтерфейси `epair`, призначає IP-адреси (схема 10.x.y.z) та налаштовує правила NAT через `ipfw` для вихідного з'єднання та опціональної переадресації вхідних портів.

4. **Файлова система** — Використовує bind-монтування `nullfs` для спільних каталогів/файлів та `devfs` для доступу до пристроїв усередині jail. Образи контейнерів — стиснені архіви (xz).

## Порівняння можливостей

| Аспект | SandboxieCrack (Windows) | Crate (FreeBSD) |
|--------|--------------------------|------------------|
| **Базова ізоляція** | Драйвер ядра (`drv/`) + DLL-перехоплення (`dll/`) — перехоплення системних викликів | Jail-и FreeBSD через `jail_setv()` — простори імен, що забезпечуються ядром |
| **Віртуалізація файлів** | Трансляція шляхів + copy-on-write (`file.c`, `file_xlat.c`) з семантикою TRUE_PATH/COPY_PATH | Повна окрема коренева ФС (архів) + bind-монтування `nullfs` для спільного доступу |
| **Ізоляція реєстру/конфігурації** | Перенаправлення гілок реєстру (`key.c`, `key_flt.c`) з фільтрацією ключів; злиття реєстру через `key_merge.c` реалізує семантику COPY_PATH для значень реєстру | Непридатне — у FreeBSD немає реєстру; для кожного jail достатньо власного `/etc` |
| **Мережева ізоляція** | Попроцесна фільтрація WFP (`wfp.c`) + перехоплення DNS (`dns_filter.c`) | VNET + віртуальні інтерфейси `epair` + NAT через `ipfw` (`net.cpp`, `run.cpp`) |
| **Управління процесами** | Маніпуляція токенами + зниження привілеїв (`token.c`, `process.c`), рівні цілісності | Ізоляція процесів на рівні jail; `jexec` для виконання всередині jail |
| **Ізоляція IPC** | Фільтрація іменованих каналів/ALPC (`ipc.c`, `ipc_port.c`), блокування динамічних портів | Ізоляція простору імен IPC на рівні jail (контроль через `allow.sysvipc`) |
| **Ізоляція GUI** | Фільтрація класів вікон, перехоплення буфера обміну (`gui.c`, `guihook.c`) | Спільний доступ до X11-сокета через монтування `nullfs` (без ізоляції) |
| **Перехоплення API** | DLL-впровадження + IAT/inline-перехоплення (`dllhook.c`, `hook_tramp.c`) | Не потрібне — модель jail не вимагає перехоплення на рівні API |
| **Ізоляція криптографічного API** | Перехоплення DPAPI (`crypt.c`) — перехоплює `CryptProtectData`/`CryptUnprotectData` для ізоляції сховища облікових даних у пісочниці | Не потрібне — домашній каталог кожного jail ізолює файли облікових даних |
| **Зашифровані пісочниці** | Зашифровані томи ImBox через `MountManager.cpp` — AES-шифрування, підтримка як файлів образів (`type=img`), так і дисків у ОЗП (`type=ram`), форматування NTFS, призначення літер дисків | Не реалізовано |
| **Знімки** | Файлові знімки зі скремблюванням імен (`file_snapshots.c`) | Не реалізовано |
| **Обмеження ресурсів** | Обмеження кількості процесів на пісочницю | Не реалізовано (доступний фреймворк RCTL у FreeBSD) |
| **Рівень сервісів** | Сервіси на C++ з дротовим протоколом (`svc/`) — брокеринг процесів, COM-проксі, GUI-сервер | Відсутній — один бінарник виконує все |
| **Шаблони/Пресети** | Типи пісочниць: Standard, Enhanced Privacy, Data Protection, Application Compartment | Відсутні — ручне створення YAML для кожного контейнера |
| **Конфігурація** | На основі INI (`Sandboxie.ini`) з GUI-редактором та валідацією | На основі YAML (`+CRATE.SPEC`) з програмним розбором |
| **Управління пакетами** | Непридатне (запускає програми хоста) | Інтеграція з FreeBSD `pkg` — встановлення пакетів в образ контейнера |
| **Сервіси COM/D-Bus** | Перехоплення COM/DCOM (`com.c`, 101 КБ) — перехоплення `CoCreateInstance`, перенаправлення DCOM, проксіювання `COM_IUNKNOWN` | Не реалізовано (можлива ізоляція D-Bus на рівні jail) |
| **Ізоляція буфера обміну** | Перехоплення буфера обміну через GUI-перехоплення + посередництво `GuiServer` між пісочницями | Без ізоляції — спільний доступ через X11-сокет |
| **Контроль створення процесів** | Перехоплює `CreateProcessW`/`ShellExecuteEx` (`proc.c`, 100 КБ) — перенаправляє дочірні процеси в пісочницю | Власний механізм jail — `jexec` для входу, межа jail забезпечується ядром |
| **Перехоплення системних викликів** | SSDT-перехоплення 50+ системних викликів NT на рівні ядра | Не потрібне — jail-и FreeBSD забезпечують ізоляцію на рівні ядра без перехоплення системних викликів |
| **Ізоляція SCM/сервісів** | Перехоплення диспетчера служб (`scm.c`, `scm_*.c`, `serviceserver.cpp`) — ізольовані програми перелічують/запускають сервіси в контексті пісочниці; запобігає доступу до сервісів хоста | Сервіси `rc.d` на рівні jail; ізоляція сервісів jail є вбудованою |
| **Проксіювання іменованих каналів / сокетів** | Проксі-сервер іменованих каналів (`namedpipeserver.cpp`, 853 рядки) + перехоплення каналів (`file_pipe.c`) — проксіює канали через межу пісочниці | Unix domain-сокети; монтування `nullfs` окремих сокетів для контрольованого спільного доступу |
| **Ізоляція терміналу/консолі** | Перехоплення API терміналу (`terminal.c`) + привілейований термінальний сервер (`terminalserver.cpp`) — ізолює вікна консолі для кожної пісочниці | Виділення PTY на рівні jail через правила `devfs` для `/dev/pts` |
| **Проксіювання мережевого API** | Проксіювання API мережевої інформації (`netapiserver.cpp`, `iphlpserver.cpp`) — контролює перелік мережевих ресурсів та запити IP-помічника | VNET забезпечує власну ізоляцію мережевої інформації для кожного jail |
| **Захищене сховище** | Проксі-сервер PStore (`pstoreserver.cpp`) — ізолює збережені облікові дані та секрети для кожної пісочниці | Домашній каталог кожного jail; відсутнє спільне сховище облікових даних |
| **Переносимість контейнерів** | Пісочниці прив'язані до інсталяції хоста | Стиснені архіви (файли `.crate`) — портативні образи |

## Аналіз відсутніх можливостей Crate

Можливості, присутні в SandboxieCrack, але відсутні в Crate:

1. **Зашифровані контейнери** — SandboxieCrack надає зашифровані томи ImBox з AES; Crate не підтримує шифрування.

2. **Управління знімками** — SandboxieCrack підтримує знімки на рівні файлів з ланцюжками батько-нащадок; Crate не має механізму знімків.

3. **Обмеження ресурсів на контейнер** — SandboxieCrack обмежує кількість процесів на пісочницю; Crate не застосовує обмежень ресурсів (хоча фреймворк RCTL у FreeBSD міг би це забезпечити).

4. **Фільтрація DNS** — SandboxieCrack перехоплює DNS-розв'язання для кожної пісочниці зі зіставленням шаблонів доменів; Crate лише налаштовує базову NAT-мережу.

5. **Шаблони конфігурації** — SandboxieCrack пропонує попередньо визначені профілі безпеки (Privacy, Data Protection тощо); Crate вимагає ручного створення YAML.

6. **Графічний інтерфейс управління** — SandboxieCrack має повноцінний Qt GUI для управління пісочницями; Crate працює лише з командного рядка.

7. **Валідація конфігурації** — SandboxieCrack перевіряє налаштування з контекстними підказками; Crate має лише базову валідацію специфікації.

8. **Файлова система copy-on-write** — SandboxieCrack прозоро перенаправляє файлові записи до копій у пісочниці; Crate використовує повні копії кореневої ФС без COW.

9. **Ізоляція GUI/робочого столу** — SandboxieCrack фільтрує операції з вікнами, ізолює робочі столи та забезпечує посередництво GUI між пісочницями через GuiServer; Crate надає спільний доступ до X11-сокета хоста без жодної ізоляції.

10. **Ізоляція буфера обміну** — SandboxieCrack забезпечує окремі буфери обміну для кожної пісочниці з налаштовуваними політиками вставки між пісочницями; Crate не має ізоляції буфера обміну.

11. **Ізоляція сервісів COM/D-Bus** — SandboxieCrack перехоплює активацію COM та проксіює об'єкти DCOM; Crate не має ізоляції D-Bus чи шини сервісів.

12. **Ізоляція SCM/диспетчера служб** — SandboxieCrack перехоплює диспетчер служб Windows, щоб ізольовані програми могли перелічувати та керувати сервісами в контексті пісочниці; Crate покладається на вбудовану ізоляцію сервісів jail.

13. **Ізоляція терміналу/консолі** — SandboxieCrack ізолює вікна консолі та термінальні сесії для кожної пісочниці за допомогою виділеного термінального сервера; Crate має базовий PTY на рівні jail, але без явного управління термінальними сесіями.

14. **Проксіювання іменованих каналів / Unix-сокетів** — SandboxieCrack має виділений проксі-сервер іменованих каналів (853 рядки) для контрольованого обміну через канали між пісочницями; Crate не має еквівалентного механізму проксіювання Unix domain-сокетів.

## Сильні сторони кожного підходу

### Сильні сторони SandboxieCrack

- **Прозорість для застосунків** — Ізольовані програми працюють без модифікацій; перехоплення API робить пісочницю невидимою для застосунків.
- **Детальний контроль** — Політики на рівні окремих процесів, файлів, ключів реєстру. Можливість дозволу/заборони окремих викликів API.
- **Без дублювання кореневої ФС** — Спільне використання ОС хоста; копіюються лише змінені файли (copy-on-write). Мінімальне використання диска.
- **Зріла функціональність** — 20+ років розробки. Шифрування, знімки, обмеження ресурсів, фільтрація DNS, проксіювання COM.
- **Глибина інтеграції** — Глибока інтеграція з Windows: контекстні меню провідника, опція «Запустити в пісочниці», автозапуск програм.

### Сильні сторони Crate

- **Ізоляція на рівні ядра** — Jail-и є примітивом ядра. Немає перехоплень у просторі користувача, які можна обійти — міцніша межа безпеки.
- **Повна ізоляція мережевого стеку** — VNET надає кожному контейнеру власний мережевий стек, таблицю маршрутизації та файрвол. SandboxieCrack лише фільтрує на рівні WFP.
- **Переносимість** — Образи контейнерів (файли `.crate`) — самодостатні архіви. Легко розповсюджувати та відтворювати.
- **Простота** — Один бінарник, ~2000 рядків C++. Легко перевіряти та аналізувати властивості безпеки.
- **Вбудоване управління пакетами** — Використовує екосистему FreeBSD `pkg` для побудови образів контейнерів із потрібним програмним забезпеченням.
- **Скриптові перехоплення** — YAML-специфікація підтримує shell-скрипти на 10 етапах життєвого циклу (перед/після створення jail, налаштування користувача, запуск сервісів тощо).
- **Без модулів ядра** — Використовує лише стандартні засоби FreeBSD: jail/VNET/`ipfw`. Без власного коду ядра.
- **Екосистема, готова до ZFS** — Підтримка ZFS у FreeBSD дає змогу реалізувати COW, знімки та шифрування без змін у застосунках.
